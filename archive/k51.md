# HealthKit Access Issues - Exhaustive Analysis

**Date:** May 1, 2026  
**Purpose:** Thorough analysis of all potential causes for HealthKit authorization/calls getting into a bugged state

## 1. AUTHORIZATION FLOW ARCHITECTURE ISSUES

### 1.1 State Management Disconnect

**Location:** `HealthKitService.swift:82-89`, `App.swift:58-66`

```swift
// HealthKitService.init()
private init() {
    Task {
        let status = await self.authorizationRequestStatus()
        if status == .unnecessary {
            self.hasRequestedAuthorization = true
        }
    }
}
```

```swift
// App.swift task modifier
.task {
    await healthKit.synchronizeAuthorization()
    if settings.hasCompletedSetup && !healthKit.hasRequestedAuthorization {
        log.warning("Resetting onboarding because HealthKit has not been requested...")
        settings.hasCompletedSetup = false
    }
    await store.load(allowCachedSnapshot: true)
}
```

**Potential Issues:**
- **Race Condition**: `synchronizeAuthorization()` runs in `.task` but `init()` also spawns a Task. Both modify `hasRequestedAuthorization` without synchronization
- **Timing Issue**: `hasRequestedAuthorization` is initialized asynchronously and may not be set when `synchronizeAuthorization()` completes
- **Double-source-of-truth**: App relies on both `hasRequestedAuthorization` (published) and `authorizationRequestStatus()` (runtime), which can diverge

### 1.2 hasRequestedAuthorization Semantics

**Location:** `HealthKitService.swift:22, 107, 159`

```swift
@Published var hasRequestedAuthorization: Bool = false
```

- Set to `true` after `requestAuthorization()` completes (line 107)
- Set to `true` if `authorizationRequestStatus()` returns `.unnecessary` (init + sync)
- **Problem**: `.unnecessary` means "authorization was requested at some point" but does NOT mean user granted access. If user denied all permissions, this still becomes `true` and app proceeds to dashboard with no actual Health data access

### 1.3 Status Check Timing Window

**Location:** `HealthKitService.swift:112-129`

```swift
func authorizationRequestStatus() async -> HKAuthorizationRequestStatus? {
    guard HKHealthStore.isHealthDataAvailable() else { return nil }
    return await withTimeout(seconds: 5) {
        await withCheckedContinuation { continuation in
            self.store.getRequestStatusForAuthorization(toShare: [], read: self.allReadTypes) { status, error in
                // ...
            }
        }
    } fallback: {
        log.error("authorizationRequestStatus timed out")
        return nil
    }
}
```

**Potential Issues:**
- **5-second timeout** may be insufficient on slower devices or when Health app is busy
- On timeout, returns `nil`, which caller treats as "unknown" — app may proceed without proper handling
- Callback-based API (`getRequestStatusForAuthorization`) wrapped in continuation may have edge cases with HealthKit's internal queue

## 2. AUTHORIZATION TYPE SET MISMATCH

### 2.1 Authorization vs Read Types Discrepancy

**Location:** `HealthKitService.swift:67-80`

```swift
/// Types for authorization request (must be HKSampleType, excludes workoutType)
private var authorizationTypes: Set<HKSampleType> {
    var types: Set<HKSampleType> = categorySampleTypes
    types.formUnion(quantitySampleTypes)
    return types
}

/// All readable types for status checking (includes workoutType)
private var allReadTypes: Set<HKObjectType> {
    var types: Set<HKObjectType> = categorySampleTypes
    types.formUnion(quantitySampleTypes)
    types.insert(workoutType)
    return types
}
```

**Critical Issue:**
- `authorizationTypes` (what we request) ≠ `allReadTypes` (what we check status for)
- `workoutType` is `HKObjectType` not `HKSampleType`, so it's excluded from authorization request
- **Result**: `authorizationRequestStatus()` checks status for `workoutType`, which may return `.sharingDenied` even though we never requested it, causing confusion
- If system considers workoutType as "notDetermined" because never requested, status logic may be incorrect

### 2.2 Type Resolution Failures

**Location:** `HealthKitService.swift:37-48, 55-65`


**Potential Issues:**

## 11. SCENARIOS THAT CAUSE BUGGED STATE

### Scenario A: User Denies All Permissions on First Launch

1. User taps "FIND MY STREAKS" in onboarding
2. `requestAuthorization()` called
3. User denies all permissions in Health sheet
4. `requestAuthorization()` returns (no error thrown by HealthKit for denial)
5. `hasRequestedAuthorization = true` set unconditionally
6. `postStatus == .shouldRequest` check may pass (status check timing issue)
7. App proceeds to dashboard with `hasCompletedSetup = true`
8. `fetchHistory()` returns empty data (no error, just nil collections)
9. Dashboard shows "NO ACTIVE STREAKS" or cached stale data
10. User has no clear path to re-request permissions

### Scenario B: Watch Setup After iPhone Setup

1. iPhone setup complete, `hasCompletedSetup = true` in shared defaults
2. Watch app launches for first time
3. `hasRequestedAuthorization` starts `false`, but may be set `true` by init Task
4. `WatchRootView` checks both flags — initially shows onboarding
5. But if settings sync quickly, `hasCompletedSetup = true` 
6. Watch may skip onboarding before HealthKit auth requested
7. Watch has no Health access but thinks setup is complete

### Scenario C: Authorization Revoked in Settings

1. User previously granted access, app working
2. User revokes Health access in iOS Settings → Privacy & Security → Health → Streak Finder
3. App continues running with cached data
4. Next `fetchHistory()` call returns empty data (no error)
5. Streaks appear "frozen" at last known values
6. No UI indication that Health access was revoked
7. Empty state won't show because cached data exists

### Scenario D: Timeout in Status Check

1. Device under heavy load or Health app busy
2. `authorizationRequestStatus()` times out after 5 seconds
3. Returns `nil` (from fallback)
4. Calling code treats `nil` as "unknown" — may proceed with assumptions
5. In `OnboardingView.requestAuth()`, pre-check status is `nil`
6. Still proceeds to call `requestAuthorization()`
7. But subsequent post-check also times out
8. `postStatus == .shouldRequest` check never evaluates correctly
9. App state uncertain — may proceed or show error incorrectly

### Scenario E: Fresh Install with iCloud Restore

1. User has app on iPhone with `hasCompletedSetup = true`
2. User gets new device, restores from iCloud backup
3. App group UserDefaults may restore `hasCompletedSetup = true`
4. HealthKit authorization is NOT restored (per-device)
5. App launches, believes setup is complete
6. `hasRequestedAuthorization` computed as `false` (fresh install, never requested)
7. App triggers onboarding reset per `App.swift:63-66`
8. **But**: If timing issue or code path missed, app may proceed to dashboard
9. Dashboard shows empty or stale data

### 16.1 New Comprehensive Authorization State Model

**Replace boolean `hasRequestedAuthorization` with a proper state enum:**

```swift
// HealthKitService.swift - Replace lines 12-23

enum HealthKitAuthState: Equatable, Sendable {
    case unknown           // Initial state, not yet checked
    case notRequested      // Never requested (shouldRequest == .shouldRequest)
    case requestedPending    // Request made, awaiting user response
    case deniedAll         // Requested, user denied all types
    case partialAccess     // Requested, user granted some, denied some
    case fullyAuthorized   // Requested, user granted all essential types
    case unavailable       // HealthKit not available on device
    case error(Error)      // Unexpected error during check
    
    var canReadData: Bool {
        switch self {
        case .fullyAuthorized, .partialAccess: return true
        default: return false
        }
    }
    
    var shouldShowOnboarding: Bool {
        switch self {
        case .unknown, .notRequested, .deniedAll, .unavailable: return true
        case .error: return true
        default: return false
        }
    }
}

@Published var authState: HealthKitAuthState = .unknown

// Persistence key for last known state across launches
private let authStateKey = "HealthKitAuthState.lastKnown"
```

### 16.2 Replace authorizationRequestStatus() with Comprehensive Check

**Replace lines 112-129 with:**

```swift
func checkAuthorizationState() async -> HealthKitAuthState {
    guard HKHealthStore.isHealthDataAvailable() else {
        return .unavailable
    }
    
    // Use longer timeout for status check
    let status = await withTimeout(seconds: 10) {
        await withCheckedContinuation { continuation in
            self.store.getRequestStatusForAuthorization(toShare: [], read: self.allReadTypes) { status, error in
                if let error {
                    log.error("getRequestStatusForAuthorization error: \(String(describing: error))")
                    continuation.resume(returning: nil)
                } else {
                    continuation.resume(returning: status)
                }
            }
        }
    } fallback: {
        log.error("getRequestStatusForAuthorization timed out after 10s")
        return nil
    }
    
    guard let status else {
        return .error(HealthKitError.statusCheckTimeout)
    }
    
    switch status {
    case .shouldRequest:
        return .notRequested
        
    case .unnecessary:
        // Authorization was requested at some point - check actual per-type status
        return await evaluatePerTypeAuthorization()
        
    case .unknownRequestStatus:
        return .unknown
        
    @unknown default:
        return .unknown
    }
}

private func evaluatePerTypeAuthorization() async -> HealthKitAuthState {
    var authorized = 0
    var denied = 0
    var notDetermined = 0
    
    // Check authorization types (the ones we actually request)
    for type in authorizationTypes {
        let status = store.authorizationStatus(for: type)
        switch status {
        case .sharingAuthorized: authorized += 1
        case .sharingDenied: denied += 1
        case .notDetermined: notDetermined += 1
        @unknown default: notDetermined += 1
        }
    }
    
    log.info("Per-type auth: \(authorized) authorized, \(denied) denied, \(notDetermined) notDetermined")
    
    if authorized == 0 && denied > 0 {
        return .deniedAll
    } else if denied > 0 && authorized > 0 {
        return .partialAccess
    } else if authorized > 0 {
        return .fullyAuthorized
    } else {
        // All notDetermined but status says unnecessary - inconsistent state
        return .notRequested
    }
}
```

### 16.3 Fix requestAuthorization() with Proper Validation

**Replace lines 93-110 with:**

```swift
func requestAuthorization() async throws -> HealthKitAuthState {
    guard HKHealthStore.isHealthDataAvailable() else {
        throw HealthKitError.unavailable
    }
    
    // Pre-check
    let preState = await checkAuthorizationState()
    log.info("Requesting authorization (preState: \(preState))")
    
    // If already fully authorized, return early
    if preState == .fullyAuthorized {
        return preState
    }
    
    let types = self.authorizationTypes
    guard !types.isEmpty else {
        log.error("Authorization type set is empty")
        throw HealthKitError.noReadTypes
    }
    
    // Execute request
    do {
        try await store.requestAuthorization(toShare: [], read: types)
    } catch {
        log.error("requestAuthorization threw: \(String(describing: error))")
        throw error
    }
    
    // IMPORTANT: HealthKit's requestAuthorization doesn't return success/failure
    // We must check status AFTER the sheet closes
    // Add small delay to allow Health app to update its state
    try? await Task.sleep(for: .milliseconds(500))
    
    // Post-check
    let postState = await checkAuthorizationState()
    log.info("Authorization request complete (postState: \(postState))")
    
    // Update published state
    await MainActor.run {
        self.authState = postState
    }
    
    return postState
}
```

```swift
// Add to HealthKitService.swift

enum HealthKitQueryError: Error {
    case authorizationDenied
    case authorizationNotDetermined
    case noDataAvailable
    case underlying(Error)
}

private func verifyReadAuthorization(for type: HKObjectType) async throws {
    let status = store.authorizationStatus(for: type)
    switch status {
    case .sharingDenied:
        throw HealthKitQueryError.authorizationDenied
    case .notDetermined:
        throw HealthKitQueryError.authorizationNotDetermined
    case .sharingAuthorized:
        return
    @unknown default:
        throw HealthKitQueryError.authorizationNotDetermined
    }
}

// Modify quantityDaily() (lines 270-303)
private func quantityDaily(
    _ id: HKQuantityTypeIdentifier,
    unit: HKUnit,
    start: Date,
    end: Date
) async throws -> [Date: Double] {
    let type = HKQuantityType(id)
    
    // Verify authorization before querying
    try await verifyReadAuthorization(for: type)
    
    // Rest of implementation remains similar...
    let interval = DateComponents(day: 1)
    let predicate = HKQuery.predicateForSamples(withStart: start, end: end, options: .strictStartDate)
    
    return try await withCheckedThrowingContinuation { continuation in
        let query = HKStatisticsCollectionQuery(
            quantityType: type,
            quantitySamplePredicate: predicate,
            options: .cumulativeSum,
            anchorDate: start,
            intervalComponents: interval
        )
        query.initialResultsHandler = { _, collection, error in
            if let error {
                continuation.resume(throwing: error)
                return
            }
            
            // Treat nil collection as auth error, not empty data
            guard let collection = collection else {

            }
```
