## рџ”ґ BLOCKERS вЂ” App Store Rejection Risk

### 1. Missing `ITSAppUsesNonExemptEncryption` in widget targets

**Files:** `FitnessStreaksWidget/Info.plist`, `FitnessStreaksWatchWidget/Info.plist`

Both widget extension Info.plists are missing `<key>ITSAppUsesNonExemptEncryption</key><false/>`. Apple's export compliance scanner checks every binary in the archive. This can trigger an ITS-encryption rejection even if the main app has it.

**Fix:** Add the key to both files (copy from the iOS or watch Info.plist).

---

### 2. `HKWorkoutType` excluded from HealthKit authorization

**File:** `Shared/Services/HealthKitService.swift` вЂ” `authorizationTypes` array

Per `docs/g51.md`, `HKWorkoutType` is incorrectly excluded from `authorizationTypes` because it's treated as a non-`HKSampleType`. This means workout-based streaks (per-workout-type, workout count, workout minutes) can never read data.

If any prod user tracks workouts as a streak metric, those streaks will be empty.

**Fix:** Add `HKWorkoutType.workoutType()` to the read authorization types.

---

### 3. `App.swift` resets onboarding on every cold launch if HealthKit status check fails

**File:** `FitnessStreaks/App.swift`

Per `docs/g51.md`, when the HealthKit authorization status check returns `.notDetermined` (which it does for read-only types even after authorization), `hasCompletedSetup` gets set back to `false` destructively. This forces legit users through onboarding again on every cold start.

**Fix:** Only reset `hasCompletedSetup` when the no-data path is taken (e.g., zero readable types authorized), not on transient status failures. Or check `HKHealthStore.authorizationStatus(for:)` only for write types, not read types.

## рџџ  HIGH вЂ” Should Fix Before Submission

### 4. GitHub Pages URLs must be verified live

**URLs in code:**
- Privacy policy: `https://jackwallner.github.io/fitness-streaks/privacy-policy.html` (in `OnboardingView.swift`, `ProPaywallView.swift`, `SettingsView.swift`)
- Support page: `https://jackwallner.github.io/fitness-streaks/` (in App Store Connect metadata)

App Review checks liveness. If GitHub Pages is down or the repo was renamed, the policy link 404s and the reviewer rejects.

**Fix:** Load both URLs from a device on cellular (no Wi-Fi/cache) before submission. Confirm they return 200.

### 5. No privacy policy link on onboarding intro screen

**File:** `FitnessStreaks/Views/OnboardingView.swift`

The intro screen (first thing a reviewer sees) has no direct privacy policy link. The policy is only accessible from the privacy panel sub-screen later in onboarding and from Settings.

**Fix:** Add a small `Link("Privacy Policy", destination: ...)` below the "CONNECT APPLE HEALTH" button on the intro screen.

---

### 6. Watch flashes onboarding on every cold start

**File:** `FitnessStreaksWatch/WatchRootView.swift`

Per `docs/g51.md`, the watch app briefly shows the onboarding view before snapping to the dashboard on every cold start. This is visible as a flash and looks like a bug.

**Fix:** Read the `hasCompletedSetup` flag from App Group UserDefaults before deciding which view to show, not after an async check.

---

### 7. Stand Hours not auto-selected in onboarding core metrics

**File:** `FitnessStreaks/Views/OnboardingView.swift`

Stand Hours is one of the most universal HealthKit metrics (every Apple Watch user has it), but it's not in the auto-selected "core" set during onboarding. Users have to manually add it.

**Fix:** Add `HKQuantityType(.appleStandTime)` to the initial `trackedStreaks` set in the onboarding flow.

---

### 8. `StreakStore.load()` runs discovery engine twice

**File:** `Shared/Services/StreakStore.swift`

Per `docs/fs-issues-428.md` item 2.1, `load()` triggers two discovery passes instead of one вЂ” doubling HealthKit queries and loading time.

**Fix:** Consolidate the two discovery calls into one.

## рџџЎ MEDIUM вЂ” Polishes Worth Doing

### 9. `commitThresholds` is write-once вЂ” thresholds never update as fitness improves

**File:** `Shared/Services/StreakEngine.swift`

Once a threshold is committed for a streak, it never recalibrates. A user who progresses from 5K to 10K steps will never have their streak intensity updated to match.

**Fix:** Add a periodic recalibration trigger (e.g., once per month or on user request) that updates thresholds.

---

### 10. Steps goal not rounded to hundreds

Shows `7737` not `7700`. Looks unfinished.

**Fix:** Round `discoverBestThreshold` output to the nearest 100 for steps, 5 for exercise minutes, etc.

---

### 11. Heart rate query is scientifically inaccurate

**File:** `Shared/Services/HealthKitService.swift`

`totalCalories` calculation uses an `Г—1.4` arbitrary multiplier. The value shown is misleading.

**Fix:** Either remove the calories metric or fetch actual active energy from `HKQuantityType(.activeEnergyBurned)`.

---

### 12. SwiftData has no migration strategy + `DailyActivity` lacks unique constraint

**Files:** `Shared/Services/DataService.swift`, model definitions

If the schema changes in a future version, SwiftData will silently destroy the store. Duplicate rows can accumulate.

**Fix:** Add a unique constraint on `DailyActivity` (date + metric composite key) and define a `SchemaMigrationPlan`.
