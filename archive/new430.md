# Fitness Streaks v430 Implementation Plan

**Date:** April 30, 2026  
**Goal:** Polish the app for App Store release by fixing 10 identified UX issues  
**Approach:** First-time user perspective audit + targeted fixes

---

## Executive Summary

This plan addresses 10 polish issues identified through real-world usage testing. The issues span onboarding flow, goal display clarity, heatmap visualization, recalibration feedback, and visual consistency. Each fix is designed to make the app more intuitive for first-time users while preserving the retro pixel aesthetic.

## Issues & Implementation Details

### Issue 1: Stand Hours Not Auto-Selected, Order Not Right

**Current Behavior:**
- Onboarding pre-selects only 4 "core" metrics: `[.steps, .activeEnergy, .exerciseMinutes, .workouts]`
- Stand Hours (`.standHours`) is excluded from auto-selection despite being a fundamental Apple Health metric
- In `StreakPicker.swift`, the same `coreMetrics` array determines sorting priority

**User Impact:**
First-time users who care about Stand Hours must manually find and select it, creating friction for a metric that Apple actively promotes via the Watch and Activity rings.

**Root Cause:**
`OnboardingView.swift:462` and `StreakPicker.swift:14` define:
```swift
private let coreMetrics: [StreakMetric] = [.steps, .activeEnergy, .exerciseMinutes, .workouts]
```

**Proposed Fix:**
1. Add `.standHours` to `coreMetrics` in both locations
2. Consider ordering: steps, exercise, stand, activeEnergy, workouts mirrors Apple's Activity ring priority

**Implementation:**
```swift
// OnboardingView.swift:462 and StreakPicker.swift:14
private let coreMetrics: [StreakMetric] = [.steps, .exerciseMinutes, .standHours, .activeEnergy, .workouts]
```

**Testing:**
- Fresh install onboarding should pre-select Stand Hours if user has data
- Stand Hours should appear near top of picker list

### Issue 2: Percentage Should Supplement Goal, Not Replace It

**Current Behavior:**
- `StreakHero.goalLine` shows: `"of 10,000 steps goal"` (awkward phrasing)
- The status area shows either "LOCKED" or "45% COMPLETE" 
- When incomplete, the actual goal number is de-emphasized

**User Impact:**
Users see "45% COMPLETE" but must mentally calculate what the goal actually is. The phrase "of X goal" is grammatically awkward.

**Root Cause:**
- `StreakHero.swift:86-88`: `goalLine` concatenates awkwardly
- `StreakHero.swift:102-105`: `statusText` replaces goal info with percentage

**Proposed Fix:**
Restructure the display to always show current value vs goal, with percentage as supplemental info:
- Current: `6,234`
- Goal: `of 10,000 steps (62%)` 
- Streak: `12 day streak`

**Implementation:**
```swift
// StreakHero.swift - goalLine becomes more natural
private var goalLine: String {
    let pct = Int(min(1, streak.currentUnitProgress) * 100)
    return "\(streak.format(currentUnitValue: streak.threshold)) \(streak.unitLabel) · \(pct)%"
}

// statusText simplified to just show LOCKED state
private var statusText: String {
    streak.currentUnitCompleted ? "LOCKED" : ""
}
```

**Testing:**
- Verify goal line reads naturally: "10000 steps · 62%" not "of 10000 steps goal"
- Verify percentage updates in real-time as HealthKit data changes

### Issue 3: "Today's Goal X of Y of Goal" Is Weird

**Current Behavior:**
`goalLine` produces: `"of 10000 steps goal"` - the word "goal" appears twice implicitly ("today's goal" context + "steps goal").

**User Impact:**
Awkward phrasing makes the app feel unpolished. Users notice grammatical oddities.

**Root Cause:**
`StreakHero.swift:86-88`:
```swift
private var goalLine: String {
    "of \(streak.format(currentUnitValue: streak.threshold)) \(streak.unitLabel) goal"
}
```

The word "goal" is hardcoded while the context (from `goalContext`) already says "today's goal".

**Proposed Fix:**
Remove "goal" from goalLine; rely on context from goalContext or restructure entirely:
- Option A: `"Goal: 10,000 steps (62%)"`
- Option B: `"10,000 steps · 62% toward goal"`
- Option C: `"6,234 / 10,000 steps"`

**Recommendation:** Option C is clearest - shows current/target with unit once.

**Implementation:**
```swift
private var goalLine: String {
    let goal = streak.format(currentUnitValue: streak.threshold)
    let pct = Int(min(1, streak.currentUnitProgress) * 100)
    return "Goal: \(goal) \(streak.unitLabel) · \(pct)%"
}
```

### Issue 4: Heatmap Lookback Should Use Actual Date, Not Separate Option

**Current Behavior:**
`HeatmapDateRange` enum includes `.lookbackPeriod` as a separate option. The heatmap picker shows: [30d] [90d] [6mo] [LOOKBACK] [1yr]

**User Impact:**
Users don't understand what "LOOKBACK" means without reading the Settings label. It's confusing to have a setting drive a view option.

**Root Cause:**
`CalendarHeatmap.swift:8`:
```swift
enum HeatmapDateRange: String, CaseIterable {
    case last30Days = "30d"
    case last90Days = "90d"
    case last180Days = "6mo"
    case lookbackPeriod = "LOOKBACK"  // Confusing
    case fullYear = "1yr"
}
```

**Proposed Fix:**
Remove `.lookbackPeriod` case. The heatmap should default to showing data from the lookback date (based on user's setting), but labeled clearly. Instead, add specific day counts that align with common lookback settings.

**Implementation:**
```swift
enum HeatmapDateRange: String, CaseIterable {
    case last30Days = "30d"
    case last90Days = "90d"
    case last180Days = "6mo"
    case last365Days = "1yr"  // Renamed from fullYear for consistency
    
    func days(lookbackDays: Int) -> Int {
        switch self {
        case .last30Days: return 30
        case .last90Days: return 90
        case .last180Days: return 180
        case .last365Days: return 365
        }
    }
}
```

Default selection should be based on `settings.lookbackDays`:
```swift
// In StreakDetailView init or onAppear
private func defaultHeatmapRange() -> HeatmapDateRange {
    switch settings.lookbackDays {
    case 0...45: return .last30Days
    case 46...135: return .last90Days
    case 136...270: return .last180Days
    default: return .last365Days
    }
}
```

### Issue 5: Heatmap Should Expand to Use Space When Not 1 Year

**Current Behavior:**
- Fixed cell size: `private let cell: CGFloat = 8`
- Fixed gap: `private let gap: CGFloat = 2`
- 365 days = ~52 columns = ~520pt width
- 30 days = ~5 columns = ~50pt width (tiny!)

**User Impact:**
When viewing 30d or 90d range, the heatmap is a tiny strip instead of using available screen width. Looks broken.

**Root Cause:**
`CalendarHeatmap.swift:40-41` hardcodes cell dimensions without considering available space.

**Proposed Fix:**
Calculate cell size dynamically based on available width and number of weeks to display. Use `GeometryReader` to measure container width.

**Implementation:**
```swift
struct CalendarHeatmap: View {
    // Remove fixed cell/gap constants
    @State private var containerWidth: CGFloat = 0
    
    private func cellSize(for weeks: Int) -> CGFloat {
        let gap: CGFloat = 2
        let available = containerWidth - CGFloat(weeks - 1) * gap
        return max(4, min(12, available / CGFloat(weeks)))  // Clamp between 4-12pt
    }
    
    var body: some View {
        GeometryReader { geo in
            // ... existing code using calculated cell size
        }
    }
}
```

Alternative: Use `ScaleToFit` or allow cells to expand proportionally:
```swift
// Each week column takes equal width
VStack(spacing: gap) {
    ForEach(0..<7, id: \.self) { wd in
        cellView(day: day)
            .frame(maxWidth: .infinity, maxHeight: .infinity)
    }
}
.frame(maxWidth: .infinity)
```

**Testing:**
- 30d view: cells should be ~20-30pt each
- 365d view: cells should shrink to fit, ~6-8pt each
- Verify heatmap fills width of card on all range selections

### Issue 6: Primary Font Color Mismatch on Home Page

**Current Behavior:**
- `StreakHero.swift:39-41`: Current value uses `accent` (lime when complete, magenta when not)
- `StreakHero.swift:17`: Title uses `streak.metric.accent` (metric-specific color)
- `StreakDetailView.swift:91-93`: Uses `streak.currentUnitCompleted ? Theme.retroLime : streak.metric.accent`

**User Impact:**
On dashboard, incomplete streak shows magenta number but title uses metric color (e.g., blue for steps). In detail view, number matches metric color. Inconsistent.

**Root Cause:**
`StreakHero` uses a computed `accent` that changes based on completion state, overriding the metric color.

**Proposed Fix:**
Make `StreakHero` match `StreakDetailView` behavior:
- When complete: lime green (consistent across all metrics)
- When incomplete: metric's accent color

**Implementation:**
```swift
// StreakHero.swift - remove the computed accent that ignores metric color
private var valueColor: Color {
    streak.currentUnitCompleted ? Theme.retroLime : streak.metric.accent
}

// Use valueColor for currentValue text
Text(currentValue)
    .foregroundStyle(valueColor)
```

**Testing:**
- Steps streak (incomplete): number should be blue like title
- Steps streak (complete): number should be lime green
- Verify consistency between dashboard hero and detail view

### Issue 7: Recalibration Text Never Shows "Complete, New Goal is X"

**Current Behavior:**
1. User taps Recalibrate
2. `actionMessage = "Recalibrating from Apple Health..."` (line 271)
3. Store reloads
4. `onChange` detects `!isLoading`, sets `actionMessage = "Recalibrated."` (line 39)
5. Message shows generic "Recalibrated." without the new goal value

**User Impact:**
Users don't know what their new goal is after recalibration. The feedback is incomplete.

**Root Cause:**
`StreakDetailView.swift:37-40`:
```swift
.onChange(of: store.isLoading) { _, isLoading in
    if !isLoading, actionMessage?.contains("Recalibrat") == true {
        actionMessage = "Recalibrated."  // No goal info!
    }
}
```

The view has no access to the "new" goal because `streak` is passed in as `let`, not observed from store.

**Proposed Fix:**
Access the refreshed streak from `store.streaks` after reload, find matching streak by trackingKey, display its threshold.

**Implementation:**
```swift
.onChange(of: store.isLoading) { _, isLoading in
    if !isLoading, actionMessage?.contains("Recalibrat") == true {
        // Find the recalibrated streak in the refreshed store data
        let refreshedStreak = store.streaks.first { $0.trackingKey == streak.trackingKey }
        let newGoal = refreshedStreak?.thresholdLabel ?? streak.thresholdLabel
        actionMessage = "Recalibration complete. New goal: \(newGoal)"
    }
}
```

**Testing:**
- Recalibrate a streak
- Verify message shows: "Recalibration complete. New goal: 8000 steps"
- Message should update after load finishes

### Issue 8: Percentage May Not Update After Recalibration

**Current Behavior:**
`streak` is declared as `let streak: Streak` at the top of `StreakDetailView`. When store reloads, the view doesn't automatically update because `streak` is not observed.

**User Impact:**
After recalibration, the UI may still show old goal/old percentage until user dismisses and re-opens detail view.

**Root Cause:**
```swift
struct StreakDetailView: View {
    let streak: Streak  // Not @State or @ObservedObject
    // ...
}
```

**Proposed Fix:**
Compute the displayed streak from the store's current data instead of relying on the passed-in value. Use the passed streak for initial identification, then look up current data.

**Implementation:**
```swift
struct StreakDetailView: View {
    let initialStreak: Streak  // Rename for clarity
    
    private var streak: Streak {
        // Prefer fresh data from store if available
        store.streaks.first { $0.trackingKey == initialStreak.trackingKey } ?? initialStreak
    }
    // ... rest of view uses self.streak
}
```

Or use `@State` and update it when store changes:
```swift
@State private var streak: Streak

.onChange(of: store.streaks) { _, _ in
    if let updated = store.streaks.first(where: { $0.trackingKey == streak.trackingKey }) {
        streak = updated
    }
}
```

**Testing:**
- Open detail view
- Recalibrate
- Verify UI updates to show new goal without dismissing view

### Issue 9: Recalibration Immediately After Onboarding Changes Goal

**Current Behavior:**
- Onboarding discovers streaks and calls `settings.commitThresholds(for: filtered)` (line 178 in StreakStore)
- If user immediately recalibrates, the committed threshold is cleared
- Re-discovery runs with potentially different data or interpretation

**User Impact:**
User asks: "Why did my goal change right after I set it up?" - feels like a bug.

**Root Cause Analysis:**
1. Onboarding: `StreakEngine.discover()` finds best threshold for user's history
2. `commitThresholds()` saves these as committed values
3. Recalibration: `clearCommittedThreshold()` removes the commitment
4. `StreakEngine.discover()` re-runs - but now `settings.lookbackDays` may differ from discovery defaults

Actually, looking closer: onboarding uses `settings.lookbackDays` (default 30), same as recalibration. The difference may be:
- Onboarding fetches fresh HealthKit data
- Recalibration may have slightly different data timing
- Race conditions or caching

**Most Likely Cause:**
User confusion - they don't understand that "recalibrate" means "re-calculate from my recent history." If their recent history is borderline, the threshold might shift slightly.

**Proposed Fix:**
Add education, not code change. In the recalibrate alert, clarify what will happen:

Current alert message:
> "Are you sure? This will update your goal based on recent activity. If the new goal is higher, you might lose your streak."

Improved message:
> "This re-analyzes your last \(settings.lookbackDays) days of activity and may suggest a different goal. If the new goal is higher than your current \(streak.thresholdLabel), your streak may break."

Also: Consider adding a "Why did this change?" explanation in settings or documentation.

**Testing:**
- Verify alert shows lookback days
- Verify user understands the 30-day analysis window

### Issue 10: Primary Card Has Excessive White Space

**Current Behavior:**
`StreakHero.swift` structure:
- Title row (metric name + icon)
- Current value + goal line
- Streak info + percentage + progress bar
- Padding: `.padding(.horizontal, 14)` + `.padding(.vertical, 16)`

**User Impact:**
Card feels "bizarre" with too much empty space, especially on larger phones or when metric name is short.

**Root Cause:**
- Fixed padding creates inconsistent visual density
- `VStack(alignment: .leading, spacing: 14)` may be too loose
- `Spacer(minLength: 0)` in some HStacks doesn't actually expand

**Proposed Fix:**
Tighten spacing and center content vertically within the card:

```swift
var body: some View {
    VStack(alignment: .leading, spacing: 10) {  // Reduced from 14
        // Header row
        HStack(alignment: .top, spacing: 12) {
            VStack(alignment: .leading, spacing: 2) {  // Tighter
                Text(titleText)
                Text(goalContext.uppercased())
            }
            Spacer()  // Push icon to right
            Image(systemName: streak.displaySymbol)
        }
        
        // Value section - add flexible spacer to center
        Spacer(minLength: 4)  // Push content toward vertical center
        
        VStack(alignment: .leading, spacing: 2) {
            Text(currentValue)
            Text(goalLine)
        }
        
        Spacer(minLength: 4)
        
        // Bottom section
        VStack(alignment: .leading, spacing: 4) {
            HStack {
                Text(streakLine)
                Spacer()
                Text(statusText)
            }
            PixelProgressBar(...)
        }
    }
    .padding(.horizontal, 14)
    .padding(.vertical, 14)  // Reduced from 16
    .frame(maxWidth: .infinity, minHeight: 180, alignment: .center)  // Center content
}
```

Alternative: Make the card compact and let content breathe naturally, or make it taller and more prominent.

**Recommendation:** Add `minHeight` and center alignment to distribute space intentionally rather than having it cluster at top.

**Testing:**
- View on iPhone 14 Pro, iPhone SE, iPad
- Verify card doesn't look cramped or sparse on any device
- Verify progress bar has adequate touch target size
