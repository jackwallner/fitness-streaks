### 1.2 hasRequestedAuthorization Semantics - **CONFIRMED**

**Location:** `HealthKitService.swift:85-86, 107, 159`

```swift
// Line 85-86 (init)
if status == .unnecessary {
    self.hasRequestedAuthorization = true  // Set even if all denied!
}

// Line 107 (requestAuthorization)
hasRequestedAuthorization = true  // Unconditional

// Line 159 (synchronizeAuthorization)
hasRequestedAuthorization = (status == .unnecessary)
```

**Verification:** `.unnecessary` means "user has been asked" NOT "user has granted". If user denies all permissions, status is still `.unnecessary`.

**Severity:** P0 - Boolean is misleading and causes wrong app state decisions.

## 2. PARTIALLY CONFIRMED / NUANCED ISSUES

### 2.1 Race Condition in Init vs synchronizeAuthorization - **PARTIALLY CONFIRMED**

**Claim:** `init()` Task and `.task` modifier race to set `hasRequestedAuthorization`

**Analysis:** 
- `init()` Task (lines 82-89) and `synchronizeAuthorization()` (line 59) both modify state
- BUT: `init()` only sets `true` if status == `.unnecessary`
- `synchronizeAuthorization()` (lines 154-162) does the SAME check
- Both would set the same value in most cases

**Verdict:** Race exists but is benign (both set same value). Not a critical issue.

---

### 2.2 Status Check Timeout Issue - **OVERSTATED**

**Claim:** 5-second timeout may be insufficient and causes bugged state

**Analysis:**
- Timeout returns `nil` which callers treat as "unknown"
- In practice, `getRequestStatusForAuthorization` is fast (< 100ms typically)
- Timeout only triggers under severe system stress
- App continues to work with `nil` → treated as needing auth request

**Verdict:** Real edge case but not a primary cause of bugged state. Timeout is appropriate.

---

### 2.3 Type Resolution Failures - **UNLIKELY IN PRACTICE**

**Claim:** Type resolution "may fail silently if HealthKit is in a weird state"

**Analysis:**
- Type resolution uses `HKObjectType.categoryType(forIdentifier:)` which returns nil only for invalid/unrecognized identifiers
- App uses standard Apple identifiers (`.mindfulSession`, `.sleepAnalysis`, etc.)
- These are compile-time constants that always resolve on iOS 14+

**Verdict:** Theoretical concern. Would require corrupted iOS installation to manifest.

---

### 2.4 Multi-Device iCloud Sync Issue - **MISIDENTIFIED MECHANISM**

**Claim:** `hasCompletedSetup` syncs via iCloud causing Watch issues

**Analysis:**
- `hasCompletedSetup` is stored in `UserDefaults(suiteName: appGroupID)` (line 282, StreakSettings.swift)
- App group UserDefaults are **NOT** synced via iCloud
- App group data is shared between app and extensions on SAME device only
- Watch has separate app group container

**Verdict:** Issue exists but mechanism is different. Watch and iPhone have separate settings. Problem is Watch app relying on `hasCompletedSetup` without verifying its own HealthKit auth.

## 3. ISSUES MISSED IN k51.md

### 3.1 No Per-Type Authorization Check After Request

**Gap:** k51.md mentions this but doesn't emphasize it as a root cause.

**Location:** `OnboardingView.swift:537`

```swift
if postStatus == .shouldRequest {
    errorText = "Health access needed..."
}
```

**Problem:** This only checks if iOS thinks we need to request. It doesn't check if user DENIED permissions. If user denies all, status is `.unnecessary` (already requested), not `.shouldRequest`. This check never triggers on denial.

**Impact:** User can deny all permissions, app proceeds to dashboard with no data access.
