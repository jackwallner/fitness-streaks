# App Store Review Critique — FitnessStreaks (ar428)

**Date:** 2026-04-28  
**Scope:** iOS app, watchOS app, iOS widget, watchOS complication, shared frameworks  
**Review posture:** App Store human + automated policy scanner  
**Verdict at a glance:** High-quality code, but **accessibility blindness** and **unbounded HealthKit queries** are near-certain rejection vectors. Several smaller data-integrity and metadata gaps remain.

---

## Executive Summary

The app is well-architected (SwiftData, SwiftUI, BGTaskScheduler, App Group sharing), and most of the issues from the previous `bug-analysis.md` pass have been remediated. However, an App Store reviewer in 2026 will run VoiceOver, load the app on a device with years of HealthKit data, and inspect every extension target for privacy manifests. This review finds **2 critical rejection risks**, **4 high-severity issues**, and **6 medium-severity issues** that should be resolved before submission.

## Critical — Likely Rejection

### CR-1: Zero accessibility annotations (VoiceOver / Dynamic Type)

- **Severity:** Critical  
- **Guidelines:** 4.0 (Design), 4.1 (No accessibility), 5.1.1 (User experience)  
- **Files:** Every view file (`DashboardView.swift`, `StreakDetailView.swift`, `SettingsView.swift`, `OnboardingView.swift`, `WatchTodayView.swift`, all components)  
- **Issue:** Not a single SwiftUI accessibility modifier exists in the codebase (`accessibilityLabel`, `accessibilityValue`, `accessibilityHint`, `accessibilityHidden`). The entire UI is custom retro-styled rectangles, pixel fonts (`RetroFont.pixel(8)` through `RetroFont.mono(12)`), and image-only buttons. A VoiceOver user hears nothing meaningful on the dashboard hero, the at-risk banner, the progress bar, the calendar heatmap, or the streak badges.  
- **Expected behavior:**  
  - Every `Button` must have an `accessibilityLabel` describing what it does.  
  - The hero streak card must expose `accessibilityLabel("Steps streak: 42 days, threshold 10,000 steps. Today 80% complete.")` or equivalent.  
  - `PixelProgressBar` (and any custom progress indicator) must expose `accessibilityValue("80 percent")` and `accessibilityProgressView`.  
  - `CalendarHeatmap` cells must expose hit/miss status per day.  
  - The threshold ladder rows must label the tier and streak length.  
  - `PixelToggle` must expose `accessibilityValue("On") / ("Off")`.  
  - All font sizes must respect Dynamic Type scaling (`@ScaledMetric` or `DynamicTypeSize`). Fixed `pixel(8)` text will not scale.  
- **Remediation:** Add an accessibility pass to every view. Use `.accessibilityElement(children: .combine)` on grouped cards. Replace fixed-size typography with `DynamicTypeSize`-relative sizing where possible, or at minimum add `.accessibilityLabel` to every interactive element.

### CR-2: Unbounded heart-rate sample query can crash on data-rich devices

- **Severity:** Critical (Performance / Watchdog)  
- **Guidelines:** 2.1 (Performance), 2.3.1 (No excessive resource usage)  
- **File:** `Shared/Services/HealthKitService.swift:384-430`  
- **Issue:** `heartRateMinutesAbove` executes an `HKSampleQuery` with `limit: HKObjectQueryNoLimit` over up to **400 days** with no predicate narrowing, date-interval batching, or quantity-type culling. A user with an Apple Watch can easily have **2–5 million** heart-rate samples in that window. Loading them all into an array at once causes:  
  1. Memory pressure / OOM crash.  
  2. Watchdog termination during background refresh (`BGAppRefreshTask`).  
  3. Severe UI jank on the main thread because the query callback runs on a HealthKit serial queue but the `withCheckedThrowingContinuation` resumes on the caller context.  
- **Expected behavior:** The query must use a **quantity-type predicate** (e.g., `HKQuery.predicateForQuantitySamples(...)` if available), a **batched / anchored query**, or at minimum a **sampling / statistical aggregation** instead of enumerating every raw sample. If raw enumeration is truly required, cap the per-day sample window or use `HKStatisticsCollectionQuery` with `HKQuantityTypeIdentifier.heartRate` and `.discreteAverage` to down-sample before client-side filtering.  
- **Remediation:** Refactor `heartRateMinutesAbove` to use `HKStatisticsCollectionQuery` with 1-minute or 5-minute buckets, or apply a **quantity sample predicate** that filters to samples above the BPM threshold in HealthKit before returning results. If raw sample enumeration is still required, process in `HKAnchoredObjectQuery` batches rather than `HKSampleQuery` with `NoLimit`.

## High — Strong Risk of Rejection or 1.0-star Review

### HI-1: Force-unwrap crash on launch if SwiftData fails in-memory fallback

- **Severity:** High  
- **Guidelines:** 2.1 (Performance — crashes)  
- **File:** `Shared/Services/DataService.swift:37`  
- **Issue:**  
  ```swift
  return try! ModelContainer(for: schema, configurations: [inMemory])
  ```  
  If the on-disk store is corrupt and the in-memory fallback also fails (extremely rare but possible on memory-constrained devices or during low-storage conditions), this `try!` crashes the app on launch. The user can never open the app again until reinstalled.  
- **Expected behavior:** If even in-memory SwiftData fails, return a non-persisting container or fail gracefully into a no-data state rather than crashing.  
- **Remediation:** Replace `try!` with `try?` and return an optional, or use a `do/catch` block that falls back to a simple no-op container or a fatal-safe error state.

### HI-2: Broken-streak banner uses "day" for weekly streaks

- **Severity:** High  
- **Guidelines:** 4.0 (Design — incorrect information), 4.1 (User confusion)  
- **File:** `FitnessStreaks/Views/DashboardView.swift:182`  
- **Issue:**  
  ```swift
  Text("\(broken.brokenLength)-day \(broken.metric.displayName.lowercased()) run")
  ```  
  The `BrokenStreak` model carries `cadence: StreakCadence`, but the banner hardcodes "-day" regardless of whether the streak was daily or weekly. A reviewer testing with a weekly workout streak will see "5-day workout run" — factually wrong.  
- **Expected behavior:** The banner should read `\(broken.brokenLength) \(broken.cadence.pluralLabel)` (e.g., "5 weeks", "12 days").  
- **Remediation:** Use `broken.cadence.pluralLabel` instead of the hardcoded "day" string.

### HI-3: Widget and watch-widget extensions lack `PrivacyInfo.xcprivacy` manifests

- **Severity:** High  
- **Guidelines:** 5.1.1 (Privacy — required reason APIs), automated scanner policy  
- **Files:** `FitnessStreaksWidget/` and `FitnessStreaksWatchWidget/`  
- **Issue:** Both widget targets access `UserDefaults` (via `SnapshotStore.load()` / `save()` using `UserDefaults(suiteName:)`), which is a **required reason API** under `NSPrivacyAccessedAPICategoryUserDefaults`. The main iOS app and watch app each have a `PrivacyInfo.xcprivacy` declaring `CA92.1`, but the widget and watch-widget binaries do not. Apple's automated upload scanner now checks every binary in the archive for manifests when required-reason APIs are linked.  
- **Expected behavior:** Every app-extension binary that uses `UserDefaults` must either:  
  (a) contain its own `PrivacyInfo.xcprivacy` with the same `CA92.1` declaration, or  
  (b) prove to the scanner that the API use is only in the main app binary. Because `SnapshotStore` is compiled into each extension target, the scanner will likely flag the widget/watch-widget binaries.  
- **Remediation:** Add a `PrivacyInfo.xcprivacy` file to both `FitnessStreaksWidget` and `FitnessStreaksWatchWidget` targets (or reuse the same file via a reference) declaring `NSPrivacyAccessedAPICategoryUserDefaults` with reason `CA92.1`.

### HI-4: Legacy `fetch` background mode is declared alongside modern `BGTaskScheduler`

- **Severity:** High  
- **Guidelines:** 2.5.4 (Multitasking — background modes must be justified)  
- **File:** `FitnessStreaks/Info.plist:38-41`  
- **Issue:** `UIBackgroundModes` contains `<string>fetch</string>`. The app uses `BGTaskScheduler` (`BGAppRefreshTaskRequest`) for background refresh, which is the modern iOS 13+ API. The legacy `fetch` mode is redundant and can trigger an App Store review question: "Why does your app require background fetch?" If the reviewer believes `BGTaskScheduler` is sufficient, they may reject for unnecessary background-mode entitlement.  
- **Expected behavior:** Either remove the legacy `fetch` key (if `BGAppRefreshTask` alone is sufficient), or provide a strong justification in the review notes explaining why both are needed. For a HealthKit-only streak app, `fetch` is almost certainly unnecessary.  
- **Remediation:** Remove `<string>fetch</string>` from `UIBackgroundModes` and verify that `BGAppRefreshTask` still schedules correctly on device.

## Medium — Should Fix Before Submission

### MD-1: Grace-day tier award uses all streaks, not just the hero, causing confusing tier jumps

- **Severity:** Medium  
- **File:** `Shared/Services/StreakSettings.swift:307-314`  
- **Issue:** `awardGraceDays(from:)` computes `tier = streaks.map { $0.current / 30 }.max() ?? 0` across the **entire filtered streak list** (hero + badges). A long-running secondary badge (e.g., 90-day steps) can award grace days even if the user's primary hero is only a 5-day streak. The UI shows "+1 every 30 days" but the user may not understand why a badge streak drove the award.  
- **Expected behavior:** Grace-day tiers should be tied to the hero streak, or the UI should explicitly state that any tracked streak contributes.  
- **Remediation:** Either filter to `store.hero` before computing the tier, or update the settings copy to say "+1 every 30 days across any tracked streak."

### MD-2: Watch complication timeline refreshes only hourly even though HealthKit updates continuously

- **Severity:** Medium  
- **File:** `FitnessStreaksWatchWidget/WatchComplication.swift:30-34`  
- **Issue:** `getTimeline` sets `next = Calendar.current.date(byAdding: .hour, value: 1, to: .now)`. The complication therefore updates once per hour. For a streak app where the user may complete their goal at 8:05 AM, the complication could show an outdated "at risk" state until 9:00 AM. Watch complications should reload more aggressively after significant events (e.g., after a background refresh or when the user opens the app).  
- **Expected behavior:** Use `.atEnd` or a much shorter interval (15 min) after the current hour, and trigger `WidgetCenter.shared.reloadAllTimelines()` from the watch app after `store.load()` completes.  
- **Remediation:** Change timeline policy to `.after(min(nextHour, now + 15*60))` during active hours, and add `WidgetCenter.shared.reloadAllTimelines()` calls inside the watch app's `handleBackgroundRefresh` after `store.load()`.

### MD-3: Privacy policy & support URLs are hardcoded and could 404

- **Severity:** Medium  
- **Guidelines:** 5.1.1 (Privacy — privacy policy required)  
- **File:** `FitnessStreaks/Views/SettingsView.swift:394-440`  
- **Issue:** The app links to `https://jackwallner.github.io/fitness-streaks/privacy-policy.html` and `https://jackwallner.github.io/fitness-streaks/`. If these GitHub Pages deployments are ever unpublished, renamed, or rate-limited, the app will present a 404. App Store Review checks that privacy-policy URLs are reachable at submission time.  
- **Expected behavior:** URLs must be live and stable for the lifetime of the app version.  
- **Remediation:** Verify both URLs are reachable today. Optionally add a local fallback or a `SFSafariViewController` with an error banner if the site is unreachable, and monitor uptime.

### MD-4: Notification copy for broken streak uses vague "Restart anytime" without clarifying data loss

- **Severity:** Medium  
- **Guidelines:** 5.1.1 (Transparency), 4.0 (Clarity)  
- **File:** `Shared/Services/NotificationService.swift:104-105`  
- **Issue:** The broken-streak notification says: `"Your \(metric) streak ended at \(length). Restart anytime — the count is yours to set again."` The phrase "the count is yours to set again" implies the user can manually edit the streak length, but the app does not actually allow manual editing of historical streak counts. The user can only re-select or recalibrate thresholds.  
- **Expected behavior:** Notification copy should accurately reflect the action space: "Pick a new goal or recalibrate from Health."  
- **Remediation:** Rewrite the broken-streak notification body to match the actual in-app recovery options (`BrokenStreakSheet` offers "Keep Same Goal", "Pick New Goal", or dismiss).

### MD-5: `StreakDetailView` recalibration path omits `hourlySteps`, so hour-window streaks never recalibrate

- **Severity:** Medium  
- **File:** `FitnessStreaks/Views/StreakDetailView.swift:255-267`  
- **Issue:** `suggestedThreshold` calls `StreakEngine.discover(...)` without passing `hourlySteps:`. Therefore, if the user views the detail of an hour-window streak and taps **Recalibrate**, the engine cannot rediscover the hour-window threshold and will likely suggest a whole-day threshold instead (or none at all).  
- **Expected behavior:** Recalibrating an hour-window streak should re-run discovery with the same `hourlySteps` data used during the initial load.  
- **Remediation:** Pass `hourlySteps: store.hourlySteps` (or equivalent cached hourly data) into the `StreakEngine.discover` call inside `suggestedThreshold`.

### MD-6: `DashboardView.atRiskBanner` appears for every badge simultaneously if all are incomplete

- **Severity:** Medium  
- **File:** `FitnessStreaks/Views/DashboardView.swift:224-226`  
- **Issue:** Inside `badgeGrid`, the code shows:  
  ```swift
  if shouldShowAtRisk, !streak.currentUnitCompleted, streak.current >= 2 {
      atRiskBanner(for: streak)
  }
  ```  
  `shouldShowAtRisk` is a global boolean based on the notification time. If the user has 6 badges and none are completed after the reminder threshold, the dashboard will show **6 red at-risk banners** stacked vertically. This is visually noisy and could overwhelm the user.  
- **Expected behavior:** Limit at-risk banners to the hero only, or to the single most urgent badge (highest current streak + lowest progress).  
- **Remediation:** Either remove the per-badge at-risk banner entirely (hero-only risk is sufficient), or compute a single `mostUrgentBadge` and display only that banner.

## Low — Polish Before Submission

### LO-1: `UIRequiredDeviceCapabilities` restricts to `healthkit`, effectively making the app iPhone-only

- **Severity:** Low  
- **File:** `FitnessStreaks/Info.plist:46-49`  
- **Issue:** `UIRequiredDeviceCapabilities` contains `<string>healthkit</string>`. While iPadOS 17+ supports HealthKit, this key prevents installation on older iPads and may signal to review that iPad is not supported. Apple increasingly encourages universal apps.  
- **Expected behavior:** If the app is intentionally iPhone-only, no action needed. If iPad support is desired, test on iPad and remove the capability key (the app will simply show an empty state if HealthKit is unavailable).

---

### LO-2: No `WKBackgroundModes` declaration in watch Info.plist

- **Severity:** Low  
- **File:** `FitnessStreaksWatch/Info.plist`  
- **Issue:** The watch app schedules background refreshes via `WKApplication.shared().scheduleBackgroundRefresh(...)` but does not declare `WKBackgroundModes` (e.g., `workoutProcessing` or `backgroundFetch`). In watchOS 10+, background refresh often works without explicit background modes for HealthKit-linked apps, but some Watch builds may warn.  
- **Expected behavior:** If background refresh works reliably during TestFlight testing, no action is needed. Otherwise, add the appropriate `WKBackgroundModes` key.

---

### LO-3: `DateHelpers.shortFormatter` uses `MMMd` template without locale override

- **Severity:** Low  
- **File:** `Shared/Utilities/DateHelpers.swift:41-45`  
- **Issue:** `shortFormatter` uses `setLocalizedDateFormatFromTemplate("MMMd")` which is locale-aware, but the rest of the app uses hardcoded English labels ("DAYS IN A ROW", "THRESHOLD TIERS", etc.). A user in a non-English locale will see mixed localized and English text.  
- **Expected behavior:** Either localize all strings (properly), or force the formatter to English to match the retro arcade aesthetic consistently.  
- **Remediation:** Set `shortFormatter.locale = Locale(identifier: "en_US_POSIX")` for consistency, or add `Localizable.strings` for all UI copy.

---

### LO-4: Widget small view uses gradient background that may violate widget design guidelines

- **Severity:** Low  
- **File:** `FitnessStreaksWidget/FitnessStreaksWidget.swift:138-172`  
- **Issue:** The `systemSmall` widget applies a full `LinearGradient` background (`Theme.streakGradient`). Apple recommends that widgets use the system's `WidgetBackground` or `containerBackground` materials so they adapt to the user's home-screen wallpaper and iOS tinting. A bold gradient can look out of place on some wallpapers.  
- **Expected behavior:** Use `containerBackground(.fill.tertiary, for: .widget)` or similar system material, applying the gradient only to a small accent strip or the hero number.  
- **Remediation:** Replace the full gradient fill with a system container background and a subtle gradient accent on the number or icon.

## Build & Test Readiness Checklist

Before submitting to App Store Connect, complete the following:

- [ ] **Accessibility pass** — Run VoiceOver on every screen. Add `.accessibilityLabel` to every button, card, and custom control. Test with **Large Text** enabled in Accessibility settings.  
- [ ] **Heart-rate query stress test** — Install on a device with 2+ years of Apple Watch heart-rate data. Verify `store.load()` completes in < 3 seconds without memory spike.  
- [ ] **Privacy manifest coverage** — Ensure all 4 targets (iOS app, watch app, iOS widget, watch widget) contain a `PrivacyInfo.xcprivacy` with `CA92.1`.  
- [ ] **Background mode audit** — Remove `fetch` from `UIBackgroundModes` if `BGAppRefreshTask` is sufficient. Document justification in review notes if retained.  
- [ ] **Broken-streak copy fix** — Change `DashboardView` banner to use `broken.cadence.pluralLabel`.  
- [ ] **DataService crash fix** — Replace `try!` with graceful fallback.  
- [ ] **URL liveness check** — Confirm privacy-policy and support GitHub Pages are reachable from a cellular connection (no Wi-Fi).  
- [ ] **iPad smoke test** — Run on iPad (if universal) or confirm iPhone-only intent.  
- [ ] **Widget stress test** — Add widget to home screen and lock screen; verify it updates within 15 minutes of completing a streak goal.  
- [ ] **Recalibration test** — Create a custom hour-window streak, break it, then tap Recalibrate and verify the suggested threshold is still hour-window based.  
- [ ] **Grace-day tier clarity** — Decide whether grace days should key off hero-only or all streaks; update UI copy accordingly.  
- [ ] **At-risk banner noise** — Decide whether badges should show at-risk banners; if not, remove the per-badge check.
